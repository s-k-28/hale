# HALE — Apple App Review Gap Audit: Final Report

## Verdict

HALE is **not submittable today**, but the distance to submittable is short and well-mapped. The single hard blocker is that the link domain `go.haleapp.com` is an unpurchased, parked domain — every in-app privacy/terms link, the planned Support URL, the referral universal links, and the support email address are all dead, which is a guaranteed rejection on first load. Beyond that, seven likely-rejection items remain, and notably all but one are quick code fixes: a fake purchase button on the fallback paywall, a paid "widgets" benefit that doesn't exist in the binary, an unmoderated public-squad-name UGC surface, one retrievable pro-e-cigarette RAG entry, "anonymous analytics" copy that contradicts the privacy label, and an unverified 14-day trial promise. The app's core compliance posture is genuinely strong — the community moderation pipeline, IAP-only monetization, consent-gated AI flows, push discipline, and data-security architecture all verified clean — so this is a punch-list problem, not an architecture problem. Plan roughly: 1–2 days of code fixes, plus the domain/ASC ops work, before submission.

## Code fixes to make now

### Likely rejection — fix before submission (cheapest first)

1. **Flip the NIDA pro-e-cigarette corpus entry to referenceOnly** — *1.4.3* — `knowledge/corpus.ts:681` has `"referenceOnly": false` on the NIDA Volkow blog entry ("e-cigarettes… outperform nicotine replacement therapy"; "NHS recommends e-cigarettes as quit aids"), retrievable as prioritized evidence for vaping queries (`convex/sageKnowledge.ts:46`, `convex/model/sage.prompt.ts:128`). Fix: flip to `true` and re-run ingest so the live RAG index picks up the flag (`convex/rag.ts:71`).

2. **Remove "Home-screen widgets" as a paid benefit** — *4.2 / 2.1 / 3.1.2* — `src/app/paywall.tsx:71-74` sells widgets in BENEFITS and `src/app/(tabs)/you.tsx:319-341` shows a paid LockedFeature card, but no WidgetKit extension exists anywhere (`ios/` has only the OneSignal NSE). A paying reviewer unlocks a mock card. Fix: delete the BENEFITS row and remove/relabel the You-tab widgets block (follow the `sos.tsx:313` precedent of hiding unshipped features); re-add when WidgetKit ships.

3. **Stop calling identified analytics "anonymous"** — *2.3.1 / 5.1.2(iii)* — `src/app/(onboarding)/welcome.tsx:89` ("anonymous usage analytics") and `src/app/(tabs)/you.tsx:510,519` ("Share anonymous analytics") contradict `posthog.identify(userId, …)` at `src/lib/analytics.ts:161-163` and the Linked=true privacy manifest (`app.json:58-80`). Fix: reword to "usage analytics linked to your account ID — you can turn this off anytime"; never use "anonymous" for identified analytics in-app or in ASC metadata.

4. **Fail fast on missing Convex URL instead of silent placeholder** — *2.1(a)* — `src/app/_layout.tsx:38` falls back to `https://placeholder.convex.cloud` when `EXPO_PUBLIC_CONVEX_URL` is empty (`src/lib/config.ts:6`); a misbuilt binary would pass CI but be dead on arrival for a reviewer. Fix: in production builds, throw/assert at startup when the env var is absent (pair with the EAS env-var ops step below).

5. **Make the fallback paywall's "START HALE+" button actually purchase** — *3.1.1 / 2.1(b)* — `src/app/paywall.tsx:200,284` wires the primary CTA under the "$79.99/yr · 14-day free trial" block to `onMaybeLater` (dismiss). This fallback is production-reachable: `src/lib/paywall.ts:66-70` returns NOT_PRESENTED on *any* RC error. Fix: have the CTA call `Purchases.getOfferings()` + `Purchases.purchasePackage(...)` (or re-invoke `RevenueCatUI.presentPaywall()`), reserving dismiss-only behavior for builds where `has('revenueCatIosKey')` is false. While in this file, also replace the hardcoded `$79.99` / `$6.67/mo` strings (`paywall.tsx:181-183,196-197`) with the StoreKit `product.priceString` (*3.1.2*, non-US storefronts get the wrong price otherwise).

6. **Moderate, gate, and make reportable public squad names** — *1.2* — `convex/squads.ts:83-124` `createSquad` accepts raw `name` with no validation, no moderation, and no `bannedAt`/rules-accepted writeGate (unlike `communityPosts.ts:103-108`); `publicSquads` (squads.ts:271-281) serves these names to all users via `src/app/squads.tsx`. A community-banned user can publish an offensive name to a public surface. Fix: (a) reuse the writeGate in createSquad, (b) trim/validate the name and route through `internal.communityModeration.classify` (new targetType `squadName`, hide from publicSquads until published) or a synchronous blocklist, (c) extend `reportContent`'s targetType union and add a long-press Report action on Discover rows.

20. **Fix the deletion cascade for squads** — *5.1.1(v)* — `convex/users.ts:215-318` purges `squadMembers` but never `squads`: owned squads survive with the user-authored name and dangling `ownerId`, and `memberCount` is never decremented. (Permission-seeking is a legal/ops step below.)
